# 8917-expert-panel

## 定位

规范型 + Pipeline skill。把「从专家库动态选 N 个 → 并行派遣(反锚定)→ 综合」固化成一个可重复动作。

与固定 4 角色的 `council` 的**唯一区别**:专家从 `~/.claude/agents/` 的 232 个里**按议题动态选**,而非写死角色。

选角策略、三模式输出格式、subagent 模板、综合护栏的**完整细节见 [REFERENCE.md](./REFERENCE.md)**。

## 三模式(一句话)

| 模式 | 像什么 | 产出 |
|---|---|---|
| **讨论** | 圆桌会 | 各抒己见 → 共识 + 分歧点 + 各方理由 |
| **评论** | 评审会(对抗) | 挑方案问题 → 分 severity 清单,**强制含一个唱反调** |
| **规划** | 作战会议 | 分领域规划 → 带分工的总计划 |

## 七步流程

1. **抽议题** — 一句话锁定:这要决定 / 评 / 规划什么 + 成功标准。议题模糊就先问一句。
2. **定模式** — 用户指定就用;没指定 → 按议题推断,并一句话确认(讨论 / 评论 / 规划)。
3. **选专家** — 从 232 库选 **3-5 个视角最互补**的(选角策略见 REFERENCE §1)。
4. **确认名单** — 列「建议召集 X / Y / Z + 每个的理由」,用户增删换人、拍板再派。**(用户说 `--auto` 或"直接派"则跳过此步)**
5. **并行派遣** — 每个专家用 **Agent 工具起 fresh subagent**,只给议题 + 精简上下文 + 角色(**反锚定**,见下),按模式要求结构化产出。一条消息内并行发起。
6. **综合** — 带偏见护栏(REFERENCE §3),按模式输出三种形态之一。
7. **紧凑输出** — 手机可读;原始各方立场在 verdict 之前可见,不藏分歧。

## 反锚定铁律(借 council)

- subagent **只拿**议题 + 必要上下文 + 角色提示,**绝不给完整对话历史**——否则专家会被我的倾向带偏,多视角就退化成回声。
- 派遣前,我**先写下自己的初判**(立场 + 最强理由 + 最大风险),再读专家产出;综合时不简单镜像专家,也不无理由否定专家。

## 边界(不可违反)

- **默认人在环**:第 4 步选完先给名单确认;只有用户显式 `--auto` 才直接派。
- **起步只做单轮并行**。"加一轮交锋"(专家互看后再反驳一轮)是**预留开关**,用户显式要求才做,不默认上。
- subagent 是**通用 persona、零记忆**,产出是"专业初稿 / 多视角原料",**不是定论**——落地交 8917 角色(Forge/Inspector)或指挥官把关。
- 不自动 push / 发布 / 改生产;不替用户做不可逆决策。
- 专家选不到合适的(议题太窄/太私域)→ 如实说"库里没有贴合的专家,建议我直接答或换 N 个近似的",不硬凑。
